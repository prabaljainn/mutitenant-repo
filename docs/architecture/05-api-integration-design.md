# API Integration & API Key Management

## 1. Overview

The platform must be open for external integrations — third-party systems should be able to call our APIs using **API keys**. API keys can be:

- **Tenant-scoped** — generated per tenant, access only that tenant's data
- **Global** — generated by super admins, cross-tenant access (for internal microservices, data pipelines, etc.)

---

## 2. Two Authentication Mechanisms

| Method | Use Case | Token Type |
|--------|----------|------------|
| **JWT (User Auth)** | Admin dashboard, Tenant platform, human users | Short-lived access token + refresh token |
| **API Key** | External integrations, server-to-server, automated scripts | Long-lived key with optional expiry |

Both methods resolve to the same `SecurityContext` internally — API keys are translated into a principal with tenant + role info, just like JWTs.

---

## 3. API Key Data Model

```
┌─────────────────────────────────────────────────────┐
│                     api_keys                         │
├─────────────────────────────────────────────────────┤
│  _id: ObjectId                                       │
│  name: "Acme Integration Key"     (human-readable)   │
│  keyHash: "sha256:..."        (hashed, never stored  │
│                                 in plain text)        │
│  keyPrefix: "mt_sk_a3b8..."   (first 8 chars, for    │
│                                 identification)       │
│  scope: "TENANT" | "GLOBAL"                          │
│  tenantId: ObjectId | null    (null for GLOBAL keys)  │
│  permissions: ["READ", "WRITE"] | ["*"]               │
│  allowedEndpoints: ["/api/v1/**"] | ["*"]  (optional) │
│  allowedIPs: ["10.0.0.0/8"] | ["*"]       (optional) │
│  rateLimit: 1000                (req/hour, 0=unlimited)│
│  status: "ACTIVE" | "REVOKED"                         │
│  expiresAt: Date | null       (null = no expiry)      │
│  lastUsedAt: Date                                     │
│  createdBy: ObjectId          (user who created it)    │
│  createdAt / updatedAt                                │
└─────────────────────────────────────────────────────┘
```

### Indexes
```javascript
db.api_keys.createIndex({ "keyHash": 1 }, { unique: true });
db.api_keys.createIndex({ "tenantId": 1, "status": 1 });
db.api_keys.createIndex({ "keyPrefix": 1 });
db.api_keys.createIndex({ "expiresAt": 1 }, { expireAfterSeconds: 0 });  // TTL index
```

## 4. API Key Lifecycle

### 4.1 Key Generation

```
Admin Dashboard → POST /admin/api/api-keys
{
  "name": "Acme CRM Integration",
  "scope": "TENANT",
  "tenantId": "tenant_abc",
  "permissions": ["READ", "WRITE"],
  "allowedEndpoints": ["/api/v1/users/**", "/api/v1/data/**"],
  "rateLimit": 500,
  "expiresAt": "2027-01-01T00:00:00Z"   // optional
}

Response:
{
  "id": "key_id",
  "name": "Acme CRM Integration",
  "key": "mt_sk_a3b8d1b60b3b4b1a9c1a1a2b3c4d5e6f"  ← SHOWN ONLY ONCE
  "keyPrefix": "mt_sk_a3",
  "scope": "TENANT",
  "tenantId": "tenant_abc"
}
```

> ⚠️ The full key is shown **only once** at creation. It is stored as a SHA-256 hash — if lost, a new key must be generated.

### 4.2 Key Usage

External systems include the API key in the request header:

```
GET /api/v1/data/sensors
X-API-Key: mt_sk_a3b8d1b60b3b4b1a9c1a1a2b3c4d5e6f
```

Or as a query parameter (less secure, for webhooks):
```
GET /api/v1/data/sensors?apiKey=mt_sk_a3b8d1b60b3b4b1a9c1a1a2b3c4d5e6f
```

### 4.3 Key Validation Flow

```
Request with X-API-Key header
       │
       ▼
┌─────────────────────┐
│ ApiKeyAuthFilter     │
│ (Spring Security)    │
│                      │
│ 1. Extract key       │
│ 2. Hash with SHA-256 │
│ 3. Lookup in MongoDB │
│ 4. Check status,     │
│    expiry, IP,       │
│    rate limit        │
│ 5. Set SecurityContext│
│    with tenant +     │
│    permissions       │
└─────────────────────┘
       │
       ▼
  Normal request processing
  (TenantInterceptor uses tenantId from API key)
```

## 5. Admin API Key Management Endpoints

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/admin/api/api-keys` | POST | Create a new API key |
| `/admin/api/api-keys` | GET | List all API keys (paginated, filterable) |
| `/admin/api/api-keys/{id}` | GET | Get key details (without the key itself) |
| `/admin/api/api-keys/{id}` | PUT | Update key name, permissions, rate limit, etc. |
| `/admin/api/api-keys/{id}/revoke` | POST | Revoke (deactivate) a key |
| `/admin/api/api-keys/{id}/regenerate` | POST | Generate a new key for the same config |
| `/admin/api/tenants/{tenantId}/api-keys` | GET | List keys for a specific tenant |

---

## 6. Security Considerations

| Concern | Mitigation |
|---------|-----------|
| Key leakage | Keys stored as SHA-256 hashes — even DB compromise doesn't expose keys |
| Key prefix | First 8 chars stored in plaintext for identification in logs/dashboards |
| Rate limiting | Per-key rate limiting via Redis sliding window counter |
| IP allow-listing | Optional `allowedIPs` CIDR ranges per key |
| Endpoint restriction | Optional `allowedEndpoints` patterns per key |
| Audit logging | Every API key usage is logged (key prefix, endpoint, timestamp, IP) |
| Key rotation | `regenerate` endpoint creates new key, old one works for 24h grace period |

## 7. Rate Limiting Implementation

```
Redis key: ratelimit:apikey:{keyPrefix}:{window}
Value: counter
TTL: window duration

Example for 500 req/hour:
  Key: ratelimit:apikey:mt_sk_a3:2026030906
  Increment on each request
  If counter > 500 → HTTP 429 Too Many Requests
  Response includes headers:
    X-RateLimit-Limit: 500
    X-RateLimit-Remaining: 342
    X-RateLimit-Reset: 1709942400
```
