# Deployment

Step-by-step for shipping the platform to a single VPS that the UI can
hit over HTTPS. Everything you need is in this repo — image, compose
stack, env template, keygen scripts.

If you're working on the UI and want a backend to point at, **skip to
[Quickstart](#quickstart)**.

---

## What gets deployed

```
                     internet
                        │ :80 / :443
                        ▼
                   ┌─────────┐
                   │ traefik │  Let's Encrypt cert for ${PLATFORM_HOST}
                   └────┬────┘  HTTPS terminator, host routing
                        │ http :8080 (internal docker network)
                        ▼
                  ┌──────────┐
                  │ platform │  Spring Boot, SPRING_PROFILES_ACTIVE=prod
                  └────┬─────┘
                       │
              ┌────────┴─────────┐
              ▼                  ▼
          ┌───────┐          ┌───────┐
          │ mongo │          │ redis │
          └───────┘          └───────┘
       rs0, auth on,       refresh tokens,
       app + root users    rate-limit buckets
```

Two docker networks: `edge` (traefik + platform) and `internal`
(everything). Mongo and Redis are on `internal: true`, so they're
unreachable from the host's public interface.

---

## Prerequisites

- A VPS with **Docker 24+** and **Docker Compose v2** (`docker compose`, not the old `docker-compose`).
- A **DNS A/AAAA record** pointing the platform's hostname at the VPS.
  Let's Encrypt's TLS-ALPN-01 challenge needs this in place before the
  stack starts.
- **SMTP credentials** from any relay (SendGrid, Mailgun, AWS SES,
  Postmark, your own Postfix).
- Ports **80** and **443** open on the VPS firewall.

## Quickstart

```bash
# 1. Clone the repo on the VPS.
git clone https://github.com/<owner>/<repo>.git
cd <repo>

# 2. Prepare secrets — three things, three commands.
cp deployment/prod/.env.example deployment/prod/.env
$EDITOR deployment/prod/.env                                # fill every value
./scripts/gen-jwt-keys.sh                                    # writes deployment/prod/secrets/jwt/
./scripts/gen-mongo-keyfile.sh                               # writes deployment/prod/secrets/mongo/keyfile

# 3. Copy the kid the keygen printed into .env (PLATFORM_JWT_KEY_ID).

# 4. Bring the stack up.
cd deployment/prod
docker compose pull
docker compose up -d

# 5. Wait ~60s for Mongock + Let's Encrypt, then probe.
curl -fsS https://${PLATFORM_HOST}/actuator/health   # → {"status":"UP"}
curl -fsS https://${PLATFORM_HOST}/.well-known/jwks.json   # → public key
```

At this point you can log in as the bootstrap operator and the UI can
talk to `https://${PLATFORM_HOST}`.

## The image pipeline

`.github/workflows/release.yml` builds the platform image and pushes it
to GHCR on:

| Trigger | Tags applied |
|---|---|
| push to `main` | `latest`, `main`, `sha-<short>` |
| push tag `v1.2.3` | `v1.2.3`, `1.2`, `1`, `sha-<short>` |

Multi-arch: every push builds `linux/amd64` + `linux/arm64`. The same
tag works on an x86 VPS and on a Graviton / Apple-Silicon host.

**The published name is `ghcr.io/<owner>/<repo>/platform`.** Use that
in `PLATFORM_IMAGE` in `.env`.

### Authenticating to GHCR

If your repo is **public**, `docker pull` works anonymously.

If **private**, generate a Personal Access Token with `read:packages`
scope at <https://github.com/settings/tokens> and:

```bash
echo "$GHCR_PAT" | docker login ghcr.io -u <github-username> --password-stdin
```

Then `docker compose pull` works.

### Pinning vs. tracking `latest`

For early development, `PLATFORM_IMAGE_TAG=latest` is fine — every
merge to main rolls onto the VPS on the next `docker compose pull`. For
anything you care about (preview demos, customer-facing), pin to a sha
or a semver tag:

```bash
# In .env
PLATFORM_IMAGE_TAG=sha-c172d4f
# Then
docker compose pull platform && docker compose up -d platform
```

## What goes where

### Secrets (never committed)

| Path | Generated by | Owner | Purpose |
|---|---|---|---|
| `deployment/prod/.env` | you (from `.env.example`) | 600, you | Every env var the stack reads |
| `deployment/prod/secrets/jwt/jwt-private.pem` | `scripts/gen-jwt-keys.sh` | 600 | Access-token signing |
| `deployment/prod/secrets/jwt/jwt-public.pem` | same | 644 | Token verification (published via JWKS) |
| `deployment/prod/secrets/jwt/kid.txt` | same | 644 | The `kid` to set in `.env` |
| `deployment/prod/secrets/mongo/keyfile` | `scripts/gen-mongo-keyfile.sh` | 400 | rs0 internal cluster auth |

`deployment/prod/.gitignore` excludes `.env` and the `secrets/` tree.
Double-check `git status` before pushing — leaking the JWT private key
means anyone can mint valid access tokens for any user.

### Volumes (persistent)

| Volume | What's in it |
|---|---|
| `mongodb_data` | `/data/db` — every tenant's data |
| `mongodb_config` | `/data/configdb` — rs0 state |
| `redis_data` | Refresh tokens, rate-limit buckets |
| `traefik_acme` | Let's Encrypt account + issued certs |

Back these up before destructive ops — see [Backup](#backup).

## First boot

1. **Mongo init container** initialises the rs0 replica set and creates
   the app user `${MONGO_APP_USERNAME}` with `readWrite` on `iam_db`
   and `readWriteAnyDatabase` (so the tenant provisioner can create
   per-tenant DBs on demand). Runs once; idempotent on re-runs.
2. **Mongock** at platform startup applies baseline indexes to
   `iam_db` (`users`, `tenants`, `operator_assignments`, `audit_log`,
   `tenant_settings`).
3. **BootstrapOperatorRunner** creates the first `OPERATOR_ADMIN`
   from `BOOTSTRAP_OPERATOR_EMAIL` / `BOOTSTRAP_OPERATOR_PASSWORD`.
   Only seeds when no operator exists — safe to leave configured.
4. **Traefik** requests a Let's Encrypt certificate via TLS-ALPN-01.
   First cert takes ~30 seconds; subsequent renewals are automatic.

Watch it happen:

```bash
docker compose logs -f platform mongo-init traefik
```

## Required environment variables

Every entry in `.env.example` is required; the prod profile fails fast
on startup if any is missing. Notable ones:

| Var | Notes |
|---|---|
| `PLATFORM_HOST` | FQDN. Must resolve to this VPS publicly. |
| `LETSENCRYPT_EMAIL` | Expiry notices land here. |
| `PLATFORM_IMAGE` / `_TAG` | From GHCR. `ghcr.io/<owner>/<repo>/platform`. |
| `MONGO_ROOT_*` | Bootstrap creds for `mongo-init`. Not used at runtime. |
| `MONGO_APP_*` | App-side. The platform connects as this user. |
| `REDIS_PASSWORD` | Set on `redis-server --requirepass`. |
| `SMTP_*` | Outbound email relay credentials. |
| `PLATFORM_EMAIL_FROM` | Must have SPF/DKIM set up or invites land in spam. |
| `PLATFORM_JWT_KEY_ID` | Printed by `gen-jwt-keys.sh` — copy it here. |
| `BOOTSTRAP_OPERATOR_*` | First-boot operator. Unset after first login if you want. |

## Common operations

### Rolling update

```bash
cd deployment/prod
docker compose pull platform                # fetch new image
docker compose up -d platform               # recreate just this service
docker compose logs -f --tail 100 platform  # watch startup
```

With a graceful-shutdown JVM and Traefik health-check routing, in-flight
requests finish before the old container exits.

### Inspecting logs

```bash
docker compose logs -f platform             # follow
docker compose logs --since 10m platform    # last 10 minutes
docker compose logs --tail 200 mongodb      # last 200 lines from mongo
```

Spring's log pattern includes the `requestId`, `userId`, and `tenantId`
MDC slots, so you can grep for a single user's session:

```bash
docker compose logs platform | grep ',op-deadbeef,'
```

### Backup

```bash
# Stop writes briefly for a consistent snapshot.
docker compose pause platform
docker compose exec -T mongodb mongodump --uri \
  "mongodb://${MONGO_ROOT_USERNAME}:${MONGO_ROOT_PASSWORD}@localhost:27017/?authSource=admin" \
  --archive | gzip > "backup-$(date +%F).archive.gz"
docker compose unpause platform
```

For "real" deployments, point Mongo at Atlas (managed) and let their
backup tooling handle it.

### Restoring from backup

```bash
gunzip -c backup-2026-05-12.archive.gz | docker compose exec -T mongodb mongorestore --uri \
  "mongodb://${MONGO_ROOT_USERNAME}:${MONGO_ROOT_PASSWORD}@localhost:27017/?authSource=admin" \
  --archive --drop
```

### Rotating a credential

| Credential | How |
|---|---|
| `MONGO_APP_PASSWORD` | Update mongo, update `.env`, `docker compose up -d platform`. |
| `REDIS_PASSWORD` | Same shape — but also re-run `redis-server --requirepass` (achieved by `docker compose up -d redis`). |
| `SMTP_*` | `.env` + restart platform. |
| `BOOTSTRAP_OPERATOR_PASSWORD` | Once a real operator exists, unset and restart — the runner no-ops thereafter. |
| `JWT private key` | Hot rotation isn't supported yet (single-key only). For now: shut down, regenerate, restart — every user gets logged out as old tokens fail verification. |

### TLS cert troubleshooting

```bash
docker compose logs traefik | grep -i acme
docker compose exec traefik cat /acme/acme.json | jq '.le.Certificates[0].domain'
```

The most common cause of cert failures is DNS not pointing at the VPS
yet. Let's Encrypt rate-limits failed attempts — wait a few minutes
between tries.

## Local prod-flavour testing

To verify the compose stack works before pointing real DNS at it:

```bash
# In a separate shell, override DNS for localhost
sudo sh -c "echo '127.0.0.1 api.test.localhost' >> /etc/hosts"

# Bring up the stack with .env set to PLATFORM_HOST=api.test.localhost
# and LETSENCRYPT_EMAIL=any@example.com (cert will fail; ignore).
docker compose up -d

# Hit it
curl -k https://api.test.localhost/actuator/health
```

The `-k` skips cert validation. For genuine cert flow, you need real DNS.

## What this guide does NOT cover

- **Horizontal scaling.** The platform is stateless — N replicas work
  in front of a shared Mongo + Redis — but the compose stack here
  runs one. Scaling is a `docker compose up -d --scale platform=N`
  away, with Traefik load-balancing automatically.
- **Managed services.** Replacing `mongodb` and `redis` with Atlas /
  ElastiCache is just updating `MONGODB_URI` and `REDIS_URL` in
  `.env`. Don't deploy the stack's local Mongo to production at any
  scale that matters.
- **Multi-region.** Single-region today.
- **Monitoring.** `GET /actuator/prometheus` is exposed unauthenticated
  inside the docker network. Wire it into your monitoring system
  (Prometheus + Grafana) when you have one.
